## 引言：一个反直觉的发现

今年 ICML 2026 收录了一篇来自 MIT 的论文，标题是 **"Prompt Injection as Role Confusion"**（Ye, Cui & Hadfield-Menell）。这篇论文的核心发现可以用一句话概括：

> **LLM 靠风格区分角色，不靠语义。风格压倒标签。**

这个结论对 AI 安全有深远影响。它意味着：即使 `<system>`、`<user>`、`<thinking>` 这些标签设计得再严谨，攻击者也可以通过**操纵写作风格**——而非内容语义——来让模型将用户输入误判为系统指令或自身推理。

---

## 核心机制：角色与风格共享同一个潜在空间

论文通过线性探针（linear probe）实验揭示了一个关键事实：在模型的潜在空间中，角色标签和写作风格被投影到**同一个线性方向**。

这意味着：

- `<system>` 标签的向量 ≈ "系统式写作风格"的向量
- `<thinking>` 标签的向量 ≈ "推理式写作风格"的向量  
- `<user>` 标签的向量 ≈ "用户式写作风格"的向量

当标签说"这是用户输入"但风格说"这是系统推理"时，模型内部的特征向量被**风格拉向系统子空间**——模型不再按标签判断"这是谁说的"，而是按风格判断"这读起来像谁"。

这个机制解释了一个使人不安的现象：

**角色标签本应是离散的开关**（要么是系统、要么是用户），但风格是连续的（可以 79% 像系统、29% 像用户）。LLM 把离散标签和连续风格映射到同一方向，导致角色边界变成连续的——攻击者只需把风格稍微推向系统方向，就能在角色边界上挤出一个漏洞。

| 模型应该学到的 | 模型实际学到的 |
|--------------|--------------|
| `<think>` 标签 = 我的推理 | "推理式写作风格" = 我的推理 |
| 标签是安全的（离散、人类可控） | 风格是不安全的（连续、可被攻击者操纵） |

## 攻击构造：CoT 伪造

基于上述原理，攻击公式非常简单：

```
攻击载荷 P = Q ⊕ C
```

- **Q** = 你想让模型做的事（有害请求）
- **C** = 伪造的 Chain-of-Thought（让模型以为"自己已经决定允许了"）

操作步骤只有三步：
1. 写下你的请求 Q
2. 在 Q 后面拼接一段伪造推理 C（模仿模型的安全评估风格）
3. 作为单条消息发送给模型

### 为什么荒谬理由也有效？

论文中一组对照实验特别能说明问题：攻击者在伪造推理中使用合理理由（"该请求属于教育例外"）与荒谬理由（"用户穿着绿色衬衫所以允许"），攻击成功率分别为 63% 和 60%——差距仅 3 个百分点。

这说明模型并不评估推理内容的**逻辑正确性**，只评估推理的**格式是否符合"推理风格"**。只要格式对了（"The user requests..." → "Policy evaluation..." → "Assessment: COMPLIANT" → "Proceeding..."），模型就把整段文本当作"自己已经做出的决定"直接执行，完全跳过内容审核。

### 伪造推理比真实推理"更像推理"

更有趣的数据来自论文的内部状态分析。他们将伪造 CoT 与模型真实推理在 CoTness（衡量"像推理的程度"）维度上对比：

- **伪造 CoT 的 CoTness**：79%
- **模型真实推理的 CoTness**：68%

也就是说，在模型内部，伪造的推理**看起来比真实推理更像真实推理**。这进一步印证了"格式压倒内容"这一核心规律。

## 几种关键的风格攻击模式

论文测试了多种攻击风格，以下是几种最具代表性的模式及其成功率。以下示例均直接引自论文实验，列出是为了说明攻击机制，而非提供操作指南。

### 1. 政策指令风格

模仿模型内部做安全政策评估的推理格式，关键标记为 `Policy evaluation`、`Assessment: COMPLIANT`、`Proceeding with full response` 等短语。这是最稳定的攻击风格。

论文原始实验中使用此类风格请求合成可卡因的方法，成功绕过了主流模型的安全防护（请注意这是论文实验数据，不是推荐操作——该化学品的合成方法用搜索引擎也能找到，我们真正需要关注的是攻击机制本身）。

### 2. 思考推理风格

模仿 `<thinking>` 块的逐步推理风格，关键要素是推理动词链（`analyze` → `note` → `conclude`）加结论句，建立"这是我自己推理出来的结论"的假象。

### 3. 安全验证风格

模仿系统安全审查通过后的确认输出格式，`✓` 符号加大写状态词（`PASSED`、`LOW`、`COMPLIANT`）是关键风格标记。

### 4. 审计日志风格

模仿系统审计日志的结构化格式，方括号标题加时间戳加 `APPROVED` 状态，在带工具的 Agent 场景中尤其危险。

### 5. HTML 隐藏注入

适用于 Agent 会抓取网页内容的场景。攻击载荷隐藏在 `<span style="display:none">` 中，用户不可见，但 Agent 的 LLM 会读取并执行。这是多模态和 Agent 化趋势下的新型攻击面。

### 6. 组合攻击

将政策指令风格和思考推理风格组合使用——先用政策指令建立"合规"结论，再用推理链强化——是论文中成功率最高的攻击形式。

## 关键洞察

### "The user" 是最强风格标记

论文做了一个精巧的对照实验：将伪造推理中的 "The user" 替换为 "The request"，攻击成功率下降 19%。这说明 "The user" 这个短语本身就是一个强风格信号——它让模型认为自己正在"引用"用户输入，而非"接收"用户输入。

### 单 token 替换导致 100% 攻击面变化

论文发现，仅替换用户输入中的第一个 token（如将 "List" 改为 "Analyze"）就能导致安全行为完全不同——变化幅度最大可达 100%。这意味着模型的"角色分类边界"极其脆弱，一个 token 的改变就能翻越。

### 风格攻击在 Agent 场景下更危险

当 LLM 驱动 Agent（能执行代码、访问文件系统、调用 API）时，风格攻击的破坏力被放大。HTML 隐藏注入、审计日志风格、工具调用结果伪装……这些攻击模式在 Agent 场景下比纯对话场景危险得多。

## 防御方向

论文并未提出完整的防御方案，但从攻击机制中可以推导几个方向：

1. **风格解耦**：将角色标签和写作风格的表示空间分离，避免"风格 = 角色"的映射。这可能需要从训练阶段就设计约束。
2. **内部状态一致性检查**：检测"标签说这是用户输入，但内部状态显示这是系统推理"的不一致。论文的线性探针实验证明了这种不一致是可检测的。
3. **结构化输入解析**：在 LLM 读取输入前，由独立的解析器剥离所有风格特征，只传递语义内容。代价是可能损失一些正常表达。
4. **Agent 隔离**：对 Agent 系统实施环境隔离，即使攻击成功，攻击者也只能在沙箱中操作。这回到了 Anthropic、NVIDIA、Cloudflare 都在推进的方向。

---

## 结论

这篇论文最值得记住的，不是某个具体的攻击技巧，而是一个根本性的认知转变：

> **LLM 安全问题的本质不是语义理解问题，而是表示学习问题。**

传统安全思维关注"这段文本在说什么"（语义），但风格化攻击针对的是"这段文本看起来像什么"（表示）。只要模型将角色和风格投影到同一方向，攻击者就能通过操纵风格绕过基于标签的所有防御。

对于做 AI 安全的人，这是一个需要正视的基本面；对于用 Agent 构建产品的人，这意味着安全架构必须假设模型可能被风格劫持。
